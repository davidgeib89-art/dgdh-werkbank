# DGDH Substrate Boundary Cut V1

Date: 2026-03-27
Status: active first cut for mission autonomy phase change
Audience: David + Codex + ChatGPT + Copilot + future mission-cell agents

## 1. Why this cut exists

DGDH is now strong enough that the next first-principles question is no longer only:

> how do we repair the current runtime?

It is also:

> what is the firm itself, and what is only the current carrier?

That boundary matters because mission-bounded self-improvement cannot mature on a self-image that lives too heavily inside one running instance, one DB, or one current product body.

## 2. Core sentence

> Paperclip is the current carrier.
> DGDH is the deeper firm truth.

This means:

- Paperclip may continue to carry real firm work
- Paperclip may evolve and even upstream-merge where useful
- but DGDH must not be ontologically reduced to the current runtime body

## 3. What DGDH is

DGDH is primarily:

- mission
- soul
- durable memory
- governance boundaries
- reviewable capability
- recoverable firm identity

These things can be expressed through Paperclip today, but they are not identical with any one Paperclip instance.

## 4. What Paperclip is in this phase

Paperclip is currently:

- the working company OS
- the main execution and governance carrier
- the current body through which DGDH runs

That makes Paperclip important.
It does not make Paperclip the full substance of the firm.

## 5. Durable firm truth vs carrier-bound state

### 5.1 Durable firm truth that should survive carrier loss

- `company-hq/CORE.md`
- `CURRENT.md`
- `MEMORY.md`
- `SOUL.md`
- `TRINITY.md`
- direct voice profiles under `company-hq/souls/`
- capability contracts under `company-hq/capabilities/`
- mission contracts under `company-hq/mission-contracts/`
- doctrinal and architectural plans that define reusable firm shape
- portable company exports and future firm-identity exports

### 5.2 Carrier-bound state that is useful but should not be the deepest truth

- live runtime process state
- local instance state
- transient DB rows without durable export
- queue occupancy, in-flight runs, adapter sessions, temporary scheduler truth

## 6. Recovery principle

If the current runtime body disappears, the firm should still be able to say:

- who it is
- what it stands for
- what its current mission mode is
- which capabilities have been durably proven
- which company structure can be re-imported
- which truths were only runtime-local and need explicit reconstruction

## 7. First implementation consequence

The first technical cut should not try to solve all mission autonomy at once.
It should create one reviewable seam where durable firm identity becomes exportable and inspectable beside the existing company portability carrier.

That cut is:

> `firm-identity-export-recovery-truth-v1`

## 8. Practical judgement

Current judgement after this cut direction:

- `main` already proves enough real firm understructure that a total throwaway would be irrational
- the next maturity step is not more repair gravity as the main mode
- the next maturity step is bounded mission autonomy on top of a clearer substrate boundary

## 9. One sentence

> DGDH must learn to preserve and describe itself as a firm whose mission, soul, memory, contracts, and proven capability survive beyond the current Paperclip runtime body.
